# Sumire 分支改动记录

这个文件记录 `reasoning-alias` 分支相对上游 `main` 额外维护的改动。

- 当前上游基线：`0.1.136`
- 当前维护分支：`reasoning-alias`
- 分支会跟随上游 rebase，同一项改动的提交哈希以后可能变化；下面使用的是当前分支上的提交哈希。

## 2026-06-10

这次把 `main` 同步到 upstream `0.1.136`，再把 `reasoning-alias` rebase 到新的 `main`。合并时继续保留 Sumire 分支功能，并接入上游 136 的后台合规门控、OpenAI/Gateway 修复和前端类型调整。

- 当前基线版本为 `backend/cmd/server/VERSION = 0.1.136`。
- 保留 Sumire 的 OpenAI reasoning alias、prompt cache 自动注入、`gpt-5.4-mini-fast`、版本徽标、账号邮箱搜索、分组账号测试、分组定时测试和 GHCR workflow。
- README 继续使用 Sumire fork 介绍页，而不是直接覆盖为上游通用 README。

## 2026-05-21

这一天先把 Sumire 分支的基础方向定了下来：模型名可以直接表达推理强度，分支也有了自己的镜像发布流程。

- 支持 `low`、`medium`、`high`、`xhigh` 这些 OpenAI 推理强度后缀，例如 `gpt-5.5-high` 会转成 `gpt-5.5`，并注入 `reasoning.effort=high`。`890af477`
- 增加 GitHub Actions，用来构建并推送 Sumire 分支的 GHCR 镜像。`6795a0d5`
- 把 GHCR 镜像 tag 统一成小写，避免大小写导致构建或拉取异常。`65c223fa`
- 给 `gpt-5.5` 增加默认 `medium` 推理强度，并让前端版本信息带上 Sumire 标识。`63c32bb6`

## 2026-05-22

这一天主要在处理 OpenAI prompt cache。重点不只是自动补参数，也包括线上排查时能看清请求有没有走到缓存逻辑。

- 优化前端版本徽标展示，并增加 `SUB2API_DEBUG_CACHE_KEYS` 缓存排查开关。`086571c0`
- 修正缓存诊断日志里的结果类型签名。`80ecc06a`
- 给 `js-cookie` 安全扫描补充例外，处理前端依赖扫描里的已知告警。`a6cba96e`
- 修正缓存诊断日志测试里的 usage 类型，让测试结构和真实 usage 更一致。`ff775cdc`
- 手动触发一次 GHCR 镜像构建，用来验证镜像工作流。`d232a205`
- 给缓存诊断增加请求体、输入内容和输入前缀哈希，方便判断两次请求是不是同一段可缓存输入。`816452ce`
- 支持自动注入 OpenAI prompt cache 参数，新增 `SUB2API_OPENAI_AUTO_PROMPT_CACHE` 和 `SUB2API_OPENAI_PROMPT_CACHE_RETENTION`。`8541acdf`
- 新增 `gpt-5.4-mini-fast` 快捷别名，默认走低推理强度和 priority service tier。`0102424a`
- 模型组导入账号时支持一起导入模型白名单。`c565966b`

- 把 h2c 配置迁移到 Go 标准库 `http.Server.Protocols` / `http.HTTP2Config`，修掉 lint 告警，同时保留原来的配置含义。`68dc474f`
- 增强 prompt cache retention 降级逻辑，同时让账号搜索可以匹配账号邮箱。`87a0bba5`
- 收窄账号邮箱搜索测试范围，让测试更聚焦。`4903189d`
- 增强集成测试 CI 失败日志，失败时给更多上下文。`8c2a4e7b`
